# Стійке відновлення стріму + коректний облік токенів

**Дата:** 2026-06-25
**Файл, що змінюється:** `src/lib/tasks/runner.ts` (стрім-цикл `consume` + оркестрація ретраїв + фіналізатор), плюс новий хелпер реконструкції повідомлень і подія realtime.

## Проблема

Коли провайдер нестабільний, хід може довго генерувати, а потім обірватися. Зараз на зависання (`StallWatchdog`, 60 с тиші) оркестратор викликає `discardPartial()` — **стирає весь `parts`**, шле клієнту `task:reset` і пере-стрімить **той самий незмінний промпт із нуля**. Уся вже згенерована відповідь викидається; при повторному зависанні — знову з нуля; при вичерпанні спроб користувач бачить порожній `PROVIDER_UNRESPONSIVE`.

Дотичні дефекти в тому ж регіоні:

1. **Транзитні помилки не ретраяться.** У `catch` ретраяться лише capability/context; звичайний обрив з'єднання / 5xx / overloaded посеред стріму валить весь хід.
2. **Неправильний облік при обриві/скасуванні.** Фіналізатор бере usage через `await result.totalUsage`. Для обірваного стріму (user cancel через `ac`, обрив) проміс падає → `usageMeta` undefined → `recordUsage` **не викликається**, хоча провайдер виставив рахунок. Недооблік на кожному скасуванні; (i)-попап без цифр.

## Цілі

- **Якість і провайдер-коректність** понад усе (явний пріоритет; час/складність не обмежують).
- Відновлювати з місця обриву, а не починати наново — і для довгої прози, і для tool-циклів.
- Працювати **універсально на всіх провайдерах** (Anthropic 4.6+/Fable, OpenAI-сумісні, OpenRouter, кастомні baseUrl), без 400.
- Рахувати токени/ціну коректно за будь-якого завершення (clean / cancel / abort / stall-out / помилка).

## Критичне провайдерне обмеження (визначає дизайн)

Trailing **assistant-prefill** видалено на сучасних моделях Anthropic (Opus 4.6/4.7/4.8, Sonnet 4.6, Fable 5) — повертає **400**, а не «дописує». Тож «підставити частковий текст як assistant і попросити модель продовжити» **не можна** — це ламає основне сімейство моделей unClaw і ненадійне на OpenAI-сумісних.

**Універсальна заміна — continuation через user-turn.** Останнє повідомлення запиту — `user`, а не assistant-prefill, тож це звичайний хід розмови → валідно скрізь.

## Дизайн

### A. Resume-by-continuation (заміна discard+restart на stall/транзитних)

При зависанні АБО транзитній помилці (з обмеженим бюджетом спроб):

1. **Зберігаємо прогрес.** НЕ стираємо `parts`. Робимо flush буферів + форс-знімок у БД, щоб клієнтський стан = серверний `parts`.

2. **Реконструюємо continuation-запит** (ефемерний, лише для цього стріму; у збережену історію не потрапляє). Новий хелпер `buildResumeMessages(parts)`:
   - Проходить `parts` по порядку:
     - `text` → у контент останнього assistant-повідомлення.
     - `tool-call` + відповідний `tool-result` → assistant(tool-call) + tool(result). **Обидва тільки парою.**
     - `reasoning` → **відкидаємо** (часткове reasoning не можна коректно продовжити; на Anthropic ще й тягне вимогу signature/ordering).
   - **Відкидаємо «висячий» tool-call без result** (незавершений крок): провайдери (Anthropic строго) вимагають, щоб кожен `tool_use` мав парний `tool_result` — інакше 400. Continuation сам пере-видасть цей крок.
   - Результат додається до `modelMessages`, далі — синтетичний user-turn (англійською — інструкція для моделі):
     > "Your previous response was cut off mid-way. Continue from exactly where it stopped — do not repeat any text you already produced, and do not mention the interruption."
   - Кешований префікс (`systemMessages` + оригінальні `modelMessages`) лишається байт-у-байт → prompt cache діє; нові лише assistant-партіал + user-continue.

3. **Reasoning вимикаємо для continuation-стріму** (`useReasoning = false` на цьому стрімі). Це усуває Anthropic-вимогу «thinking-блок перед tool_use» для реконструйованого assistant-ходу і не псує якість фінішу відповіді.

4. **Нові дельти дописуються** до наявного останнього `text`-part. Без `task:reset`.

5. **Шов (seam).** Continuation — це новий assistant-меседж, тож модель може повторити кілька слів на стику. Запобіжник `stitchOverlap`: тримаємо хвіст партіалу (останні ~500 символів); на першій continuation-text-дельті шукаємо найдовший суфікс-партіалу, що є префіксом нової дельти, і зрізаємо його. Зрізає дослівні повтори; перефраз-рестарти (рідкісні за чіткої інструкції) не чіпає.

### B. Розширення ретраю на транзитні помилки

Класифікуємо помилку (через наявний `classifyLLMError` / категорії):
- **Транзитна** (мережа / 5xx / `overloaded` / connection reset / тайм-аут шлюзу) → той самий continuation-шлях із короткою фіксованою затримкою (`delay(1000)` з `@ai-sdk/provider-utils`).
- **Stall** → continuation одразу (провайдер уже змарнував 60 с).
- **Нетранзитна** (auth, invalid request, content-policy) → кидаємо як зараз.
- **Capability / context** → лишаються на власних шляхах із `discardPartial` (там промпт *змінюється* — stripped files / trimmed context — тож continuation незастосовний).

**Єдиний бюджет відновлень** (≈3) на хід замість окремого `MAX_STALL_RETRIES`; `MAX_TASK_MS` (`ac`) — жорстка стеля.

### C. Корисний результат при повній невдачі

Коли бюджет вичерпано / дедлайн: **лишаємо вже згенерований частковий текст** + дописуємо примітку («провайдер перестав відповідати посеред відповіді»), замість стирання в порожній `PROVIDER_UNRESPONSIVE`.

### D. Живий per-step облік токенів (фікс обриву/скасування + проміжний підрахунок)

- Лічильник `liveUsage = {input, output, cached}`. У `case "finish-step"` стрім-циклу додаємо `event.usage` (per-step `LanguageModelUsage`: `inputTokens`, `inputTokenDetails.{noCacheTokens,cacheReadTokens}`, `outputTokens`). Сума per-step кроків == `totalUsage` для чистого ходу, але переживає abort (завершені кроки віддали usage до події `abort`).
- **Прибираємо залежність від `await result.totalUsage`** у фіналізаторі. `usageMeta` рахується з `liveUsage`.
- **Перекидання в `discarded`** (білінг-тільки, поза (i)-попапом): лише на capability/context/empty-discard — там, де `parts` стирається, фолдимо поточний `liveUsage` у `discarded` і обнуляємо `liveUsage`. Stall/continuation **не** перекидають (вивід зберігається у `parts`).
- Фіналізатор:
  - `usageMeta = liveUsage` (видима відповідь, попап).
  - `recordUsage` білінг = `liveUsage + discarded` (повна правда для shared-key).
  - Тепер викликається і для `cancelled`/stall-out/error, бо `liveUsage` заповнений живим обліком.
- (Опційно, окремо) проміжний usage можна стрімити клієнту для лайв-лічильника — поза рамками цього спека, але механізм відкриває таку можливість.

## Зміни в realtime/клієнті

- Stall/continuation **не** шле `task:reset`. Замість нього — форс-знімок (наявний `saveSnapshot(true)` з `streamSeq`), щоб клієнт лишався синхронним; нові дельти йдуть із `++seq` як завжди.
- Якщо реконструкція відкинула висячий tool-call, який клієнт уже показав спінером, шлемо точкову подію, що скасовує саме цей pending-крок (на основі наявної логіки tool-подій), не чіпаючи решту.

## Розбивка на одиниці

- `buildResumeMessages(parts): ModelMessage[]` — чиста, тестована окремо (вхід: `parts`; вихід: валідний масив ModelMessages без reasoning і без висячих tool-call).
- `stitchOverlap(tailOfPartial, firstDelta): string` — чиста, тестована окремо.
- `classifyTransient(err): boolean` — поверх `classifyLLMError`.
- Акумулятор `liveUsage` + правила фолду — локально у фіналі/оркестраторі.

## Тести

- `buildResumeMessages`: лише текст; текст+tool-пари; висячий tool-call зрізається; reasoning відкидається; порожні parts → порожньо.
- `stitchOverlap`: дослівний повтор зрізається; немає перекриття — нічого не зрізає; часткове перекриття.
- `classifyTransient`: 5xx/overloaded/reset → true; auth/invalid → false.
- Облік: завершені кроки → `liveUsage` правильний; capability-discard перекидає у `discarded`; cancel після N кроків → `recordUsage` отримує usage цих кроків (а не 0).
- Інтеграційно (мок-провайдер): зависання на середині прози → continuation дописує, без `task:reset`, без дублю на стику; stall у tool-циклі → продовження з останнього завершеного кроку.

## Поза рамками

- Лайв-відображення лічильника токенів у UI (механізм відкривається, але не реалізуємо тут).
- Зміна capability/context-шляхів (лишаються з `discardPartial`).
