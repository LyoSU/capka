# Phase 0: Personal AI Chat тАФ Design Spec

> **Date**: 2026-03-20
> **Status**: Approved
> **Goal**: Working AI chat with tools, accessible via browser and Telegram, zero config files

---

## 1. Overview

Phase 0 delivers a personal AI chat platform that is useful from day one. One command to start (`docker compose up` or `npm run dev`), zero `.env` files, all configuration through a web UI setup wizard.

### Success criteria

- First message to AI within 2 minutes of `docker compose up`
- Chat works with any of: OpenAI, Anthropic, OpenRouter, Ollama
- Telegram sync works bidirectionally
- MCP tools (web search, filesystem) function correctly
- PWA installable on mobile
- Dark/light theme with system detection
- Docker image < 100MB

## 3. Zero-Config System

### 3.1 Settings table

All configuration stored in `settings` table (key-value with encrypted values):

```
Key                     | Value (examples)            | Encrypted
------------------------|-----------------------------|---------
setup_complete          | "true"                      | No
auth_secret             | <auto-generated 32 bytes>   | No (is the encryption key)
admin_email             | "yuri@kness.com"            | No
platform_name           | "AntiClaw"                  | No
telegram_bot_token      | "123:ABC..."                | Yes
telegram_webhook_secret | <auto-generated>            | Yes
```

Provider-specific API keys stored in `provider_configs` table (encrypted).

### 3.2 Encryption

- `auth_secret` auto-generated on first run via `crypto.randomBytes(32)`
- Stored as hex in `settings` table (it IS the master key)
- All sensitive values encrypted with `aes-256-gcm` using auth_secret
- Encryption utility: ~30 LOC (`encrypt(value, secret)` / `decrypt(encrypted, secret)`)

### 3.3 Auth secret lifecycle

`better-auth` requires a secret for session signing. Since we store it in DB (not .env):

1. On app startup, read `auth_secret` from `settings` table
2. If not found (first run): generate via `crypto.randomBytes(32).toString('hex')`, write to DB
3. Initialize `better-auth` lazily via a factory function тАФ `getAuth()` returns a cached instance, creating it on first call with the DB-sourced secret
4. The setup wizard calls `getAuth()` which triggers initialization before creating the admin account

This avoids the chicken-and-egg: the secret exists before `better-auth` is initialized, and `better-auth` is not created at module load time.

### 3.4 First-run detection

Middleware checks `settings.setup_complete`:
- Not found тЖТ redirect to `/setup`
- Found + no session тЖТ redirect to `/login`
- Found + session тЖТ proceed normally

### 3.4 Setup wizard

3 steps, each saves to DB immediately:

1. **Create admin account** тАФ email + password тЖТ better-auth creates user + session
2. **Add LLM provider** тАФ select provider, paste API key, select default model, test connection
3. **Connect Telegram** (optional, skippable) тАФ paste bot token, test connection, set webhook

After step 2 (or 3): mark `setup_complete = true`, redirect to `/chat`.

## 4. Database Schema

### 4.1 Tables

```sql
-- Platform settings (key-value, some values encrypted)
settings (
  key          TEXT PRIMARY KEY,
  value        TEXT NOT NULL,
  is_encrypted BOOLEAN DEFAULT false,
  updated_at   TIMESTAMP DEFAULT now()
)

-- better-auth managed tables
users (id, name, email, emailVerified, image, createdAt, updatedAt)
sessions (id, expiresAt, token, createdAt, updatedAt, ipAddress, userAgent, userId)
accounts (id, accountId, providerId, userId, ...)
verifications (id, identifier, value, expiresAt, createdAt, updatedAt)

-- LLM provider configurations (per user)
provider_configs (
  id             TEXT PRIMARY KEY,
  user_id        TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  provider       TEXT NOT NULL,         -- 'openai'|'anthropic'|'openrouter'|'ollama'
  api_key        TEXT,                  -- encrypted
  base_url       TEXT,                  -- for ollama or custom endpoints
  default_model  TEXT,
  is_active      BOOLEAN DEFAULT true,
  created_at     TIMESTAMP DEFAULT now(),
  updated_at     TIMESTAMP DEFAULT now()
)

-- Chats
chats (
  id          TEXT PRIMARY KEY,
  user_id     TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title       TEXT,
  model       TEXT,                     -- last used model for this chat
  created_at  TIMESTAMP DEFAULT now(),
  updated_at  TIMESTAMP DEFAULT now()
)

-- Messages
messages (
  id                  TEXT PRIMARY KEY,
  chat_id             TEXT NOT NULL REFERENCES chats(id) ON DELETE CASCADE,
  role                TEXT NOT NULL,     -- 'user'|'assistant'|'system'|'tool'
  content             TEXT NOT NULL,
  platform            TEXT DEFAULT 'web', -- 'web'|'telegram'
  telegram_message_id INTEGER,
  metadata            JSONB,             -- tool calls, attachments, etc.
  created_at          TIMESTAMP DEFAULT now()
)

-- Telegram account linking
telegram_links (
  id              TEXT PRIMARY KEY,
  user_id         TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  telegram_user_id INTEGER NOT NULL UNIQUE,
  telegram_username TEXT,
  linked_at       TIMESTAMP DEFAULT now()
)

-- Link codes (ephemeral, for /link flow)
link_codes (
  code       TEXT PRIMARY KEY,
  user_id    TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  expires_at TIMESTAMP NOT NULL
  -- Expired codes deleted on read (check expires_at < now) or via periodic cleanup
)
```

## 5. AI Agent

### 5.1 Mastra agent configuration

Single chat agent for Phase 0:

```
Agent: chat-agent
Model: user's selected model (from provider_configs)
Tools: MCP tools (filesystem, web-search)
Memory: Mastra Memory with:
  - lastMessages: 40
  - semanticRecall: disabled (Phase 0, no vector store)
  - workingMemory: enabled (persists user facts across sessions)
  - observationalMemory: disabled (Phase 1)
Storage: @mastra/pg pointing at the same PostgreSQL database
```

Conversation history is loaded from the app's `messages` table and passed to the agent per-request. Mastra's built-in memory (working memory, last messages) is backed by the same PostgreSQL database via `@mastra/pg` storage adapter.

### 5.2 MCP tools (Phase 0 scope)

Two MCP servers:
- **filesystem**: `@modelcontextprotocol/server-filesystem` with `--root` scoped to user's data directory
- **web-search**: `@anthropic/mcp-web-search` (requires BRAVE_API_KEY or similar тАФ entered in Settings > Integrations)

MCP servers configured via UI (Settings > Integrations), stored in DB, started on-demand.

### 5.3 Provider factory

```
getModel(provider, modelId) тЖТ returns AI SDK model instance
Supported: openai, anthropic, openrouter, ollama
Provider configs loaded from DB per user
```

### 5.4 Chat API endpoint

`POST /api/chat` тАФ streaming endpoint:
1. Authenticate via better-auth session
2. Load user's active provider config
3. Save user message to DB
4. Call Mastra agent with conversation history
5. Stream response via Vercel AI SDK `streamText`
6. On complete: save assistant message to DB
7. If user has linked Telegram: optionally notify

### 5.5 Error handling

- **Provider errors** (rate limit, invalid key, timeout): surfaced as toast in web UI, error reply in Telegram. No auto-retry in Phase 0 тАФ user retries manually.
- **Invalid API key**: detected on provider test (setup wizard + settings), shown as inline form error.
- **MCP tool failure**: tool result rendered as error card in chat with the error message.
- **Unlinked Telegram user**: bot replies with instructions to link account.
- **Streaming interruption**: partial response saved to DB, user can regenerate.

## 6. Telegram Integration

### 6.1 Setup flow

1. User pastes bot token in Settings > Integrations
2. Backend validates token via `bot.api.getMe()`
3. Registers webhook: `bot.api.setWebhook(platformUrl + '/api/webhook/telegram')`
4. Token encrypted and saved to settings

### 6.2 Account linking

1. User clicks "Link Telegram" in Settings тЖТ generates 6-digit code (valid 5 min)
2. User sends `/link CODE` to bot in Telegram
3. Bot validates code тЖТ creates telegram_links record
4. Future messages from this Telegram user тЖТ routed to platform user

### 6.3 Message sync

- Telegram message тЖТ save with `platform: 'telegram'` тЖТ call agent тЖТ reply in Telegram
- Web message тЖТ save with `platform: 'web'` тЖТ call agent тЖТ stream to web
- Telegram messages route to the user's most recently updated chat, or create a new chat if none exists
- Messages from both platforms visible in web UI with platform indicator

### 6.4 Webhook endpoint

`POST /api/webhook/telegram`:
- Verify webhook secret
- Parse update via grammY
- Handle commands: `/start`, `/link CODE`, `/new` (new chat)
- Handle text: save message тЖТ agent тЖТ reply

## 7. UI Design

### 7.1 Design system

- **Font**: Geist Sans (body) + Geist Mono (code)
- **Base text**: `text-sm` (14px) everywhere
- **Colors**: oklch-based zinc neutral palette, CSS variables
- **Components**: shadcn/ui exclusively
- **Icons**: Lucide React exclusively, `h-4 w-4` standard size
- **Radius**: 8px base (`--radius: 0.5rem`)
- **Shadows**: Only `shadow-sm` on floating elements
- **Animations**: CSS transitions for hover/focus, motion for page transitions + chat message appear only

### 7.2 Layout

Desktop: Sidebar (260px) + Main content (flex-1)
- Sidebar: logo, "New Chat" button, chat list grouped by date, settings/user at bottom
- Main: header (h-12) with title + model selector + actions, content area, input area

Mobile (< md): Sidebar as Sheet, full-width content, bottom input.

### 7.3 Key components

| Component | Implementation |
|-----------|---------------|
| Sidebar | shadcn SidebarProvider + Sidebar |
| Chat messages | Custom, divide-y pattern, ReactMarkdown |
| Chat input | Textarea + send button + attach button |
| Model selector | shadcn Select, compact (h-7), in header |

| Settings | Vertical tabs layout, form with auto-save |
| Setup wizard | 3-step form with progress |
| Theme switcher | ToggleGroup (system/light/dark) |
| Toasts | sonner |
| Streaming | Cursor blink CSS animation at end of text |

### 7.4 Chat message rendering

- User messages: right-aligned, subtle `bg-muted` background
- Assistant messages: left-aligned, no background
- Tool results: collapsible card with tool icon
- Code blocks: `bg-muted`, `rounded-md`, `font-mono`, copy button
- Timestamps: `text-xs text-muted-foreground`, shown on hover
- Platform indicator: small icon for Telegram-origin messages
- Empty state: centered icon + text + CTA button (e.g., "No messages yet" + MessageSquare + "Start a conversation")
- Streaming indicator: animated dots (three-dot pulse) before first token arrives
- Inline loading: spinner inside send button during submission

### 7.5 Chat input features

- Multi-line textarea with auto-resize (shift+enter for new line, enter to send)
- File attach button (left side)
- Send button (right side, primary when input non-empty, disabled when empty)
- Stop generation button (replaces send button during streaming)
- Model selector in header (not in input area)

### 7.6 Confirmation dialog pattern

For agent actions with side effects (file write, shell command, etc.):
- Compact card below the agent message: action description + Approve/Reject buttons
- Approve = execute and show result inline
- Reject = agent acknowledges and continues without executing

### 12.4 Real-time sync strategy (Phase 0)
SSE (Server-Sent Events) for all real-time communication:
- Chat streaming: SSE via Vercel AI SDK (built-in)
- Cross-channel sync: dedicated SSE endpoint (`/api/events`) тАФ server pushes new message notifications
- Telegram тЖТ Web: webhook saves message тЖТ SSE notifies web client тЖТ client fetches new messages
- Web тЖТ Telegram: bot.sendMessage on response completion (if user has linked TG)
- Single SSE connection per client, reconnects automatically via EventSource API

### 12.5 UI language
English for all UI strings. System prompts configurable per user (Phase 1). Keeps implementation simple and library-compatible.

### 12.6 Docker image size
MCP servers (`@modelcontextprotocol/server-filesystem`, `@anthropic/mcp-web-search`) are spawned as child processes via `npx` at runtime, not bundled in the image. The 100MB target covers only the platform itself.

### 12.7 Database: PostgreSQL from day one
Skip SQLite entirely. Use PostgreSQL for everything тАФ Drizzle ORM, better-auth, Mastra Memory (`@mastra/pg` storage adapter). This avoids SQLiteтЖФLibSQL compatibility issues and eliminates a future migration. Docker Compose includes a `postgres` service. Slightly more setup but zero migration debt.
