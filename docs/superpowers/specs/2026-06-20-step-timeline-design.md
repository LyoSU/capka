# Стрічка кроків ШІ — візуальний редизайн технічних повідомлень

**Дата:** 2026-06-20
**Статус:** затверджено (візуально, через brainstorm-компаньйон)

## Проблема

Технічні кроки агента (виконання bash, читання/створення файлів, пошук, MCP, навички)
зараз рендеряться як сухі сірі рядки з гайковим ключем для всього невідомого
(`mcp__notion__search` → «Notion search» + Wrench). Аудиторія — нетехнічні
співробітники компанії; їм важить **намір** дії, а не назва інструмента.

## Рішення (затверджений напрям)

Вертикальна **стрічка кроків** у стилі Grok/Claude, але в палітрі застосунку
(тепла кремова, монохром, тонка лінія). Кожен крок:

- кружечок-бейдж `27px` з тонким бордюром і **outlined-іконкою типу дії** всередині;
- текст наміру (`muted-foreground`), запит/шлях/команда — **моноширинним** і темнішим;
- **мета праворуч** (затверджено): тривалість, «N результатів», іконки джерел;
- розкриття деталей (вивід/артефакт) **під кроком** у тихій сірій рамці;
- стани: завершено (сірий бейдж), активний (темний бейдж + спінер), помилка (стримано червоний).

Лінія тонка (`1.5px`, `--border`), без кольорової заливки. Завершені довгі ланцюжки
згортаються у шапку «Працював N с ⌄».

### Маркер дрібних кроків
Другорядні кроки (дрібне міркування) — **крихітна крапка `9px`**, **відцентрована
по лінії** так само, як бейджі (поточний мокап мав баг: крапка з'їжджала вліво —
виправити, центр маркера завжди на осі лінії `left:13px`).

## Таксономія — 6 людських категорій (за наміром, не за інструментом)

| Категорія | Іконка | Інструменти | Як читається |
|---|---|---|---|
| **Міркування** | лампочка / крапка | reasoning | «Аналізую…»; дрібні = просто крапка |
| **Файли** | file-plus / file-pen / file-text / folder | `write_file`, `str_replace`, `read_file`, `list_files` | «Створено/Відредаговано {файл}» (головні, дають артефакт); «Прочитано/Список» (другорядні) |
| **Виконання** | terminal (bash) / code (py·js) | `execute_bash`, `execute_python`, `execute_node` | «Виконано команду / код», мова дрібним; розкриття → вивід |
| **Пошук і джерела** | search / globe | web search, browse/fetch, `search_files` | «Пошук в інтернеті {запит}» → N результатів + джерела; «Переглянуто {сайт}»; «Пошук у файлах» |
| **Підключені застосунки (MCP)** | бренд-іконка конектора / plug | `mcp__<server>__<tool>` | «{Бренд} · {дія}» (Gmail, Notion, Slack, Drive…); fallback — plug + Title-case сервера |
| **Навички (Skills)** | зірочка (sparkles) | `skill` | «Навичка «{назва}»» |

**Чому так:** користувачу байдуже `execute_python` vs `execute_node` — згортаємо у
«виконання». Але **бренд застосунку** (Gmail/Notion) та **назву навички** він
упізнає — тому це окремі категорії з власними іконками.

## Архітектура реалізації

- **`src/components/chat/steps.tsx`** — `describeStep` розширюється: повертає, окрім
  `Icon/label/activeLabel/detail`, ще `category` ("reasoning"|"file"|"exec"|"search"|"mcp"|"skill"|"other"),
  опційні `meta` (напр. тривалість, кількість результатів), `minor: boolean`, і для MCP —
  `brand` (ключ конектора для іконки). Детекція MCP за префіксом `mcp__<server>__<tool>`,
  розбір `server`/`tool`. Детекція `skill` за іменем `skill` + `input.name`.
- **Новий хелпер бренд-іконок** — мапа відомих конекторів (gmail, google-drive,
  google-calendar, notion, slack, linear, github, asana, atlassian…) → inline-SVG/колір;
  fallback — lucide `Plug`/`Blocks` + Title-case назви сервера. Тримати легким.
- **`src/components/chat/message.tsx`** — `ToolGroup`/`ToolCard` переписуються у
  **рейку**: контейнер `.rail` з псевдо-лінією; кожен крок — `.step` з абсолютним
  бейджем `.node`. Мета праворуч (`.meta`), розкриття деталей під кроком (вивід через
  наявний `ToolDetails`, артефакти через `WorkspaceLinks`/file-chip). Дрібні кроки —
  `.minor` з крапкою. Зберегти існуючу логіку станів (running/error/done) і колапс.
- **`src/components/chat/task-status.tsx`, `workspace-panel.tsx`** — споживають той самий
  `describeStep`; оновити під новий повернений тип (категорія/іконка), нічого не дублюючи.
- **`messages/uk.json`, `messages/en.json`** — нові ключі під `steps` (browse/fetch вже є;
  додати mcp-/skill-/exec-варіанти) і `chat.tool` за потреби. Стиль uk — формальне «Ви»,
  іменникові статуси (див. конвенції локалізації).

## Тестування

`describeStep` — чиста функція, вже є патерн тестів (`presenter.test.ts`). Покрити:
розбір `mcp__server__tool` → категорія mcp + brand + дія; `skill` → категорія skill +
назва; унификація execute_*; web/browse евристики; minor для reasoning. Тести спершу (TDD).

## Поза обсягом (YAGNI)

- Реальні фавікони джерел із мережі (поки — кольоровий чип з літерою/бренд-SVG).
- Анімація «заливки» лінії кольором (відкинуто як зайвий шум).
- Налаштування користувачем вигляду стрічки.
