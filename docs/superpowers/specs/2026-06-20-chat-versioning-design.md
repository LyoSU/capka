# Дерево версій чату + форки

**Дата:** 2026-06-20
**Статус:** затверджено до планування
**Контекст:** апка ще не була в проді — робимо чисто, без обережних міграцій/бекфілів.

## Проблема

Зараз чат лінійний: `messages` — плаский список, упорядкований за `created_at`,
без `parentId`. Редагування/regenerate (`use-background-chat.ts` → `rerun`/
`editMessage`) виконує `DELETE FROM messages WHERE chatId AND created_at >= anchor`
— **деструктивне обрізання хвоста**. Наслідки:

- редагування **безповоротно стирає** попередню відповідь і все після неї;
- немає версіонування / перемикання між варіантами (`‹ i/N ›`);
- форків чатів немає.

## Мета

Перейти від лінійного списку до **дерева повідомлень** з одним вказівником
«активного листа» на чат. Це покриває три бажані можливості як окремі випадки
однієї операції «додай вузол / переміни активний лист»:

1. недеструктивне редагування (повернення до попередньої гілки);
2. перемикання варіантів відповідей (`‹ i/N ›`, як у ChatGPT);
3. форки чатів (копія шляху в новий чат).

## Архітектура: дерево на `parentId` + `activeLeafId`

Видимий діалог — це **ланцюг від активного листа до кореня** (і розвернути),
а не «всі рядки за часом». `created_at` лишається, але структуру діалогу більше
не визначає — лише впорядковує сиблінгів для нумерації.

### Схема (`src/lib/db/schema.ts`)

`messages`:
- `parentId text references messages(id)` — nullable; корінь чату = `null`.
- індекс `idx_messages_parent_id` на `parentId`.

`chats`:
- `activeLeafId text references messages(id)` з `ON DELETE SET NULL` — який лист
  зараз показуємо (null = порожній чат).

Інваріанти (greenfield дозволяє ввести одразу):
- рівно один корінь (`parentId IS NULL`) на чат;
- усі сиблінги мають спільного `parentId`;
- `activeLeafId` завжди вказує на лист (вузол без дітей **на активному шляху**) —
  технічно на будь-який вузол; рендер спускається до фактичного листа.

### Read-path (`src/lib/chat/presenter.ts` + `GET /api/chat`)

1. `WITH RECURSIVE` піднімається від `chats.activeLeafId` через `parentId` до
   кореня, збирає вузли, розвертає → видимий лінійний діалог.
2. Для кожного вузла шляху рахуємо `{ siblingIndex, siblingCount }`:
   сиблінги = рядки з тим самим `parentId`, порядок за `created_at`.
   Це живить стрілки `‹ i/N ›` в UI.
3. `toUIMessages` віддає на кожне повідомлення додаткові поля
   `metadata.siblingIndex`, `metadata.siblingCount`, `metadata.parentId`.

### Edit / regenerate (стають недеструктивними)

`src/hooks/use-background-chat.ts` + `POST /api/chat`:

- **Edit** user-повідомлення `M`: створюємо `M'` з `parentId = M.parentId`
  (сиблінг `M`), запускаємо відповідь асистента дитиною `M'`; `activeLeafId`
  зсувається вниз. `M` і його піддерево лишаються в БД.
- **Regenerate** відповіді `A` (на user `U`): створюємо `A'` з `parentId = U`
  (сиблінг `A`); `activeLeafId ← A'`. Нічого не стирається.

Маршрут `DELETE /api/chat/messages` (деструктивне обрізання) **видаляємо** —
його замінює логіка створення сиблінга.

### Перемикання гілок

Новий `PATCH /api/chat` (поле `activeLeafId`) або `POST /api/chat/active-leaf`:

- клієнт надсилає обраного сиблінга;
- сервер спускається до листа його піддерева (**політика: найновіша дитина на
  кожному кроці** за `created_at`);
- ставить `chats.activeLeafId` на знайдений лист; клієнт перезавантажує діалог.

### Інтеграція з runner / worker

`src/lib/tasks/runner.ts`:

- плейсхолдер асистента вставляється з `parentId = id user-повідомлення`, на яке
  відповідаємо;
- на `task:start` і на фіналізації `chats.activeLeafId ← id листа асистента`.

Перегукується з уже злитим фіксом реконсилера зомбі (`reconcileZombies`):
перерваний воркером вузол — це просто лист зі `status: "failed"`; regenerate
робить із нього сиблінга. Жодної окремої обробки не треба.

### Форки

`POST /api/chats/fork { chatId, fromMessageId }`:

- новий чат у **тому ж проєкті** (`projectId` копіюється);
- копіюємо активний шлях корінь→`fromMessageId` (включно) у нові рядки `messages`
  (нові `id`, ланцюг `parentId` перебудовано), `activeLeafId ← скопійований лист`;
- title = оригінальний + « (копія)».

Workspace спільний: sandbox у нас **per-project**, не per-chat (див.
unclaw-phase1-project-workspaces) — отже файли між форками/гілками спільні й
**не версіонуються**. Це свідоме обмеження v1.

### UI (аудиторія — нетехнічні співробітники)

- стрілки `‹ i/N ›` рендеряться **лише** на вузлах із `siblingCount > 1` —
  непомітні, поки гілок немає;
- кнопка «Дублювати з цього місця» (форк) у меню повідомлення;
- усі підписи — формальне «Ви», локалізація `uk` (див.
  unclaw-uk-localization-conventions).

### Telegram

Завжди дописує в активний лист; гілки не перемикає (TG лінійний за природою).

## Тести (інтеграційні, патерн `queue.integration.test.ts`)

- edit/regenerate створюють сиблінга, нічого не видаляючи;
- обхід активного шляху повертає правильний лінійний діалог;
- перемикання листа спускається до правильного листа (політика «найновіша дитина»);
- форк копіює шлях корінь→точка з перебудованим `parentId`;
- нумерація сиблінгів `{ siblingIndex, siblingCount }` коректна.

## Поза обсягом (YAGNI для v1)

- версіонування файлів workspace по гілках;
- перемикання гілок із Telegram;
- запам'ятовування «останньої відвіданої» дитини на вузол (беремо найновішу);
- візуалізація дерева (показуємо лише лінійний активний шлях + стрілки).

## Порядок реалізації

Одна цілісна фіча, але логічними кроками:

1. схема (`parentId`, `activeLeafId`) + read-path по активному шляху;
2. недеструктивні edit/regenerate (insert sibling, прибрати DELETE-маршрут);
3. runner ставить `parentId`/`activeLeafId`;
4. перемикання гілок (API + UI `‹ i/N ›`);
5. форки (API + кнопка).
